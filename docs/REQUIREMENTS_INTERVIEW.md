# 原始访谈记录（馆长原话）

1. 如果只做 3 个功能，我最希望是哪 3 个？为什么？

图书检索（读者端）：没有它，书就是仓库里的死物，读者无法自助找到书。

借书/还书（馆员端）：这是图书馆存在的最根本交易，必须闭环。

读者注册/验证（馆员端）：必须知道“谁”在借书，并实时校验其当前是否可借。

为什么是这3个？ 因为这是一个完整的“人→书→交易”三角闭环。没有“采编”、“预约”、“罚款统计”我们还能手工补录，但没有这3个，图书馆根本无法开门营业。

2. 系统主要使用者有哪几类？各自需要什么权限？

普通读者：仅需 查询图书 + 查看个人借阅列表（无增删改权限）。

流通馆员（核心操作者）：拥有 借书/还书 + 新增/临时修改读者信息 + 查看全部在借清单 的权限（无删除图书和财务权限）。

系统管理员（仅1人）：拥有 上架/下架图书 + 重置读者密码 + 修改借阅规则参数（如借阅天数）的权限。

3. 借阅超期你们怎么处理？禁止借书、罚款，还是两者都有？

MVP 阶段只做“禁止借书”，不做罚款。

原因： 罚款涉及“金额计算”、“缴费状态”、“支付通道”，1周内做不完且极易出Bug。我们采取硬规则：只要读者名下存在超期未还图书，系统立即拦截其新的借书申请，直到归还为止。这比罚款更能确保图书回流，且逻辑零成本。

4. 一本图书需要记录哪些信息？（MVP极限版）

书名（必填）

作者（必填）

ISBN（必填，作为唯一标识）

总库存数量（必填）

当前可借数量（必填，由借还动作自动计算，不手动维护）

书架位置/架位号（强烈建议加，否则实体馆找书如大海捞针）

5. 一个读者需要记录哪些信息？（MVP极限版）

读者ID/借阅证号（必填，系统唯一主键）

姓名（必填）

手机号（必填，用于催还通知）

当前在借数量（系统自动计算）

借阅上限（固定默认值，如3本，暂不按类型区分）
