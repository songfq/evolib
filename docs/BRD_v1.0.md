## 1. 项目背景与商业痛点

### 1.1 项目背景

当前图书馆日常运营严重依赖人工登记和手工台账，图书与读者信息分散，借阅交易效率低下，且无法实时掌握库存与借阅状态。为在最短时间内实现「从手工到系统」的数字化转型，我们提出 EvoLib MVP 方案，聚焦核心交易闭环，确保图书馆可以「开门营业」。

### 1.2 核心商业痛点

| 痛点编号 | 痛点描述 | 业务影响 |
| --- | --- | --- |
| P-01 | 图书无法被读者自助检索，查找图书完全依赖馆员经验和纸本目录 | 读者体验差，馆员咨询负担重，图书利用率低 |
| P-02 | 借还书过程依赖手工登记，库存扣减和读者借阅额度无法实时同步 | 易出错、数据滞后，无法生成准确的借阅统计 |
| P-03 | 读者信息分散在纸质登记表或 Excel 中，无法快速验证读者身份和借阅资格 | 超期图书无法自动识别，借出风险不可控 |
| P-04 | 缺少系统化的超期管控机制，图书回流周期长，影响馆藏流通效率 | 图书资产周转慢，采购成本浪费 |

## 2. 项目目标与成功标准

### 2.1 业务目标

- **目标1**：实现读者自助检索图书，将图书查找时间从平均 10 分钟缩短至 30 秒以内。
- **目标2**：建立借阅交易闭环，使馆员端借书/还书操作单笔耗时不超过 15 秒。
- **目标3**：实现读者实名注册与实时借阅资格校验，超期拦截规则自动化，确保超期图书回流率提升至 100%（即超期即拦截，直至归还）。
- **目标4**：通过数字化管理，在 MVP 上线后一周内，实现核心业务流程（检索 → 借书 → 还书 → 注册）的 100% 线上覆盖。

### 2.2 成功标准（可量化）

| 指标 | 目标值 | 测量方式 |
| --- | --- | --- |
| 检索响应时间 | ≤ 2 秒（并发 50 用户） | 性能测试 |
| 借书/还书交易成功率 | ≥ 99.5% | 系统日志统计 |
| 超期拦截准确率 | 100%（无漏拦、无误拦） | 用例测试 |
| 馆员操作培训时长 | ≤ 30 分钟 | 培训记录 |

## 3. 项目范围边界

系统交付物包含一套完整的Web操作界面，所有业务功能（登录、检索、借还书、读者注册、图书上架等）均通过浏览器访问操作。

### 3.1 在范围（In-Scope）—— MVP 必须交付

| 序号 | 功能模块 | 包含内容 |
| --- | --- | --- |
| 1 | **用户认证/登录** | 读者/馆员/管理员通过借阅证号+密码登录；颁发JWT Token（有效期8小时），所有业务接口（检索、借还书、读者管理、图书管理）均需携带Token方可访问 |
| 2 | 图书检索（读者端） | 按书名/作者/ISBN搜索；查看图书详情（库存、位置、简介） |
| 3 | 借阅管理（馆员端） | 借书（扫描读者ID+ISBN）、还书（扫描ISBN）、查看读者在借清单 |
| 4 | 读者管理（馆员端） | 注册新读者（姓名、手机号、借阅证号）、修改手机号 |
| 5 | 图书管理（管理员端） | 上架新书（书名、作者、ISBN、库存、架位号）、下架（库存置零） |
| 6 | 超期控制 | 系统自动校验超期状态，超期读者禁止借书（无罚款功能） |
| 7 | 权限管理 | 三角色：普通读者、流通馆员、系统管理员，各司其职 |

### 3.2 不在范围（Out-of-Scope）—— 明确排除

| 序号 | 事项 | 排除原因 |
| --- | --- | --- |
| 1 | 图书采编/编目 MARC 数据 | 超出 MVP，后续扩展 |
| 2 | 预约/预借功能 | 非开馆必备，可手工登记 |
| 3 | 罚款计算与支付通道 | 1 周内无法可靠实现，且易出 Bug |
| 4 | 财务报表与统计大屏 | 非交易必需，后续迭代 |
| 5 | 邮件/短信自动催还通知 | 可通过馆员手工导出手机号联系，暂不自动 |
| 6 | 移动端 App 或微信小程序 | MVP 仅提供 Web 端（响应式） |

### 3.3 假设与依赖

- **部署环境**：内部服务器或云主机，支持 Java/Node.js + MySQL/PostgreSQL。
- **操作终端**：所有馆员配备可连接系统的终端（电脑或专用扫码设备）。
- **读者标识**：读者借阅证号为唯一 ID（可由馆员现场分配或沿用旧系统编号）。
- **借阅规则**：借阅规则参数（借阅天数、借阅上限）在 MVP 中硬编码，暂不提供 UI 配置（由管理员直接修改配置文件）。
