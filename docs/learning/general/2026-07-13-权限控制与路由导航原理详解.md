## 一、Spring Security 角色判断机制

### 1.1 角色从哪里来？

角色的完整来源链路：

```
数据库 readers表
    │
    ├─► role 字段："ROLE_READER" / "ROLE_CIRCULATION" / "ROLE_ADMIN"
    │
    ├─► 登录时：AuthService.login()
    │       │
    │       ├─► JwtTokenProvider.generateToken(readerId, role)
    │       │       │
    │       │       └─► Token Payload: {"sub":"R001", "role":"ROLE_READER"}
    │       │
    │       └─► 返回 Token 给前端
    │
    └─► 请求时：JwtAuthenticationFilter.doFilterInternal()
            │
            ├─► tokenProvider.getRoleFromToken(jwt)
            │       │
            │       └─► 返回 "ROLE_READER"
            │
            ├─► new UserPrincipal(readerId, role)
            │       │
            │       └─► getAuthorities() → [SimpleGrantedAuthority("ROLE_READER")]
            │
            └─► SecurityContextHolder.getContext().setAuthentication(auth)
```

### 1.2 hasRole 的比较机制

**hasRole 的特殊处理**：`hasRole("READER")` 会自动添加 `ROLE_` 前缀：

```java
// SecurityConfig.java
.antMatchers("/api/v1/books/**").hasRole("READER")
// 实际比较的是：ROLE_READER
```

**比较发生的位置**：`FilterSecurityInterceptor` 过滤器

**比较逻辑**：

| 比较方 | 值 |
| --- | --- |
| **用户拥有的权限** | `UserPrincipal.getAuthorities()` 返回的集合，如 `[SimpleGrantedAuthority("ROLE_READER")]` |
| **需要的权限** | `hasRole("READER")` 转换后的 `"ROLE_READER"` |

```java
// Spring Security 内部逻辑（简化）
boolean hasRole = authentication.getAuthorities().stream()
    .anyMatch(authority -> authority.getAuthority().equals("ROLE_" + roleName));
```

### 1.3 hasRole vs hasAuthority

```java
// 角色带 ROLE_ 前缀时
.hasRole("READER")       // 自动转换为 ROLE_READER
.hasAuthority("ROLE_READER")  // 效果相同

// 角色不带 ROLE_ 前缀时
.hasAuthority("READER")  // 直接比较，不添加前缀
```

### 1.4 完整流程图

```
请求到达 → JwtAuthenticationFilter → FilterSecurityInterceptor → Controller
              │                          │
              ├─► 解析 Token              ├─► 读取 SecurityConfig 配置
              │     │                    │     │
              │     └─► 获取角色          │     └─► hasRole("READER") → ROLE_READER
              │                          │
              ├─► 创建 UserPrincipal     ├─► 比较：用户权限包含 ROLE_READER？
              │     │                    │     │
              │     └─► getAuthorities() │     ├─► YES → 允许访问
              │           │              │     └─► NO → 拒绝访问（403）
              │           └─► [ROLE_READER]
              │
              ├─► 注入 SecurityContext
```

## 二、权限管理方案对比

### 2.1 当前方案：写死角色判断

```java
// SecurityConfig.java
.antMatchers("/api/v1/books/**").hasRole("READER")
.antMatchers("/api/v1/admin/**").hasRole("ADMIN")
```

**优点**：
- 实现简单，代码直观
- 性能好（内存判断）
- 适合角色固定的小型系统

**缺点**：
- 不够灵活，新增角色需修改代码
- 不支持细粒度权限控制
- 不支持动态调整权限

### 2.2 改进方案：动态权限管理

#### 2.2.1 RBAC 模型

```
用户 ──多对多── 角色 ──多对多── 权限（资源+操作）
```

#### 2.2.2 数据库设计

```sql
-- 用户-角色关联表（支持多角色）
CREATE TABLE user_roles (
    user_id VARCHAR(50),
    role_id INT,
    PRIMARY KEY (user_id, role_id)
);

-- 角色表
CREATE TABLE roles (
    id INT PRIMARY KEY,
    name VARCHAR(50) UNIQUE,
    display_name VARCHAR(100),
    description VARCHAR(500)
);

-- 权限表
CREATE TABLE permissions (
    id INT PRIMARY KEY,
    resource VARCHAR(100),
    action VARCHAR(50),
    description VARCHAR(500)
);

-- 角色-权限关联表
CREATE TABLE role_permissions (
    role_id INT,
    permission_id INT,
    PRIMARY KEY (role_id, permission_id)
);
```

#### 2.2.3 代码改进

```java
// 从基于角色判断 → 基于权限判断
.antMatchers("/api/v1/books/**").hasAuthority("BOOK_READ")
.antMatchers("/api/v1/admin/**").hasAuthority("ADMIN_ACCESS")
```

```java
@Component
public class DynamicPermissionService {
    
    public Collection<GrantedAuthority> getAuthoritiesByReaderId(String readerId) {
        // 1. 查询用户的角色
        List<String> roleIds = userRoleMapper.findRoleIdsByUserId(readerId);
        
        // 2. 查询角色的权限
        List<String> permissions = rolePermissionMapper.findPermissionsByRoleIds(roleIds);
        
        // 3. 转换为 GrantedAuthority
        return permissions.stream()
            .map(SimpleGrantedAuthority::new)
            .collect(Collectors.toList());
    }
}
```

### 2.3 方案对比

| 维度 | 当前方案（写死角色） | 改进方案（动态权限） |
| --- | --- | --- |
| **灵活性** | 差，新增角色需改代码 | 好，动态配置 |
| **扩展性** | 差，权限固定 | 好，支持细粒度权限 |
| **维护成本** | 高，每次变更需部署 | 低，配置即可 |
| **适用场景** | 小型系统、角色固定 | 中大型系统、角色权限需动态调整 |
| **实现复杂度** | 简单 | 较复杂，需设计权限模块 |
| **性能** | 快（内存判断） | 需考虑缓存 |

### 2.4 为什么当前项目采用写死角色？

- **MVP 阶段**：快速验证业务，角色固定且数量少（3个）
- **代码简洁**：易于理解和维护
- **演进路径**：后续可平滑过渡到动态权限系统

## 三、Vue Router 菜单导航流程

### 3.1 整体流程图

```
用户点击左侧菜单（如"借书操作"）
    │
    ├─► ① <router-link> 触发导航
    │       │
    │       └─► URL 变为 /circulation/borrow
    │               │
    │               ├─► ② router.beforeEach 路由守卫
    │               │       │
    │               │       ├─► 检查登录状态
    │               │       ├─► 检查角色权限
    │               │       └─► 决定是否允许访问
    │               │
    │               ├─► ③ 路由匹配
    │               │       │
    │               │       └─► 匹配到 path: '/circulation/borrow'
    │               │               │
    │               │               └─► 加载组件: BorrowView.vue
    │               │
    │               ├─► ④ <router-view> 渲染组件
    │               │       │
    │               │       └─► App.vue 中的 <router-view /> 渲染 BorrowView
    │               │               │
    │               │               └─► BorrowView 被包裹在 AppLayout 中
    │               │                       │
    │               │                       └─► AppLayout 的 <slot /> 显示内容
    │               │
    │               └─► ⑤ 页面更新
    │                       │
    │                       └─► 用户在右侧看到借书操作页面
```

### 3.2 阶段详解

#### 阶段 1：点击菜单 — `<router-link>` 触发

**代码位置**：`src/components/layout/AppSidebar.vue`

```vue
<router-link 
  :class="['app-sidebar__item', { active: $route.name === 'Borrow' }]" 
  to="/circulation/borrow"
>
  借书操作
</router-link>
```

**原理**：
- `<router-link>` 本质是 `<a>` 标签，但会阻止默认跳转
- 调用 `router.push()` 更新路由状态
- URL 变为 `http://localhost:3000/circulation/borrow`

#### 阶段 2：路由守卫 — `beforeEach` 权限校验

**代码位置**：`src/router/index.js`

```javascript
router.beforeEach((to, from, next) => {
  const auth = useAuthStore();
  
  // 公开页面直接放行
  if (to.meta.auth === false) return next();
  
  // 未登录跳登录页
  if (!auth.isLoggedIn) return next('/login');
  
  // 检查角色权限
  if (to.meta.role && !auth.hasRole(to.meta.role)) {
    return next(auth.homePath);
  }
  
  next();
});
```

#### 阶段 3：路由匹配 — 找到对应的组件

**代码位置**：`src/router/index.js`

```javascript
{ 
  path: '/circulation/borrow', 
  name: 'Borrow', 
  component: () => import('@/views/circulation/BorrowView.vue'), 
  meta: { role: 'ROLE_CIRCULATION' } 
}
```

**懒加载原理**：
```javascript
component: () => import('@/views/circulation/BorrowView.vue')
// 首次访问时才加载组件，减小首屏体积
```

#### 阶段 4：`<router-view>` 渲染组件

**代码位置**：`src/App.vue`

```vue
<template>
  <router-view />
</template>
```

**原理**：
- `<router-view>` 根据当前路由渲染对应的组件
- 当前路由匹配到 `BorrowView.vue`，所以渲染为 `<BorrowView />`

#### 阶段 5：AppLayout 布局展示

**代码位置**：`src/components/layout/AppLayout.vue`

```vue
<template>
  <div class="app-layout">
    <AppSidebar />      <!-- 左侧菜单（固定） -->
    <AppHeader />       <!-- 顶部导航（固定） -->
    <main class="app-layout__main">
      <slot />          <!-- 右侧内容（动态） -->
    </main>
  </div>
</template>
```

**插槽机制**：
- `<slot />` 允许父组件向子组件传递内容
- BorrowView 的内容通过 `<slot />` 插入到右侧主内容区

### 3.3 完整代码链路

```
AppSidebar.vue: <router-link to="/circulation/borrow">
    │
    ├─► router.push('/circulation/borrow')
    │       │
    │       ├─► URL 更新为 /circulation/borrow
    │       │
    │       ├─► router.beforeEach 校验
    │       │       │
    │       │       ├─► 检查登录状态
    │       │       ├─► 检查角色权限
    │       │       └─► next()
    │       │
    │       ├─► 路由匹配: /circulation/borrow → BorrowView.vue
    │       │
    │       ├─► <router-view /> 渲染 BorrowView
    │       │       │
    │       │       └─► BorrowView 内部使用 AppLayout
    │       │               │
    │       │               └─► AppLayout 的 <slot /> 显示 BorrowView 内容
    │       │
    │       └─► 页面更新，右侧显示借书操作
```

### 3.4 菜单高亮原理

```vue
<router-link 
  :class="['app-sidebar__item', { active: $route.name === 'Borrow' }]" 
  to="/circulation/borrow"
>
  借书操作
</router-link>
```

- `$route.name` 是当前路由的名称
- 当路由为 `/circulation/borrow` 时，`$route.name === 'Borrow'` → true
- 动态添加 `active` CSS 类 → 蓝色高亮

## 四、核心知识点总结

### 4.1 Spring Security

| 技术点 | 是什么 | 为什么用 |
| --- | --- | --- |
| `hasRole()` | 角色判断方法 | 自动添加 ROLE_ 前缀，简化配置 |
| `hasAuthority()` | 权限判断方法 | 直接比较，支持细粒度权限 |
| `UserPrincipal` | 用户主体对象 | 封装用户信息和权限 |
| `SecurityContext` | 安全上下文 | 存储当前用户认证信息 |
| `JwtAuthenticationFilter` | JWT 过滤器 | 解析 Token 并注入用户信息 |
| `FilterSecurityInterceptor` | 权限拦截器 | 执行权限决策 |

### 4.2 权限管理

| 方案 | 适用场景 | 优缺点 |
| --- | --- | --- |
| 写死角色 | MVP、角色固定 | 简单但不灵活 |
| 动态权限 | 中大型系统 | 灵活但复杂 |

### 4.3 Vue Router

| 技术点 | 作用 | 代码位置 |
| --- | --- | --- |
| `<router-link>` | 路由导航组件 | `AppSidebar.vue` |
| `router.push()` | 编程式导航 | Vue Router 内部 |
| `router.beforeEach()` | 路由守卫 | `router/index.js` |
| `<router-view>` | 路由视图组件 | `App.vue` |
| `<slot />` | Vue 插槽 | `AppLayout.vue` |
| 懒加载 | 按需加载组件 | `router/index.js` |

### 4.4 布局架构

| 组件 | 作用 | 特点 |
| --- | --- | --- |
| `AppLayout` | 整体布局容器 | 包含侧边栏、头部、主内容区 |
| `AppSidebar` | 左侧菜单 | 根据角色动态显示 |
| `AppHeader` | 顶部导航 | Logo + 用户名 + 退出按钮 |
| `AppLayout__main` | 右侧内容区 | 通过 `<slot />` 动态填充 |

## 五、为什么这样设计？

### 5.1 后端权限控制

- **分层校验**：JWT 过滤器负责认证，SecurityConfig 负责授权
- **声明式配置**：一行注解完成权限校验，代码清晰
- **可扩展性**：后续可演进为动态权限系统

### 5.2 前端路由导航

- **SPA 特性**：不刷新页面，提升用户体验
- **组件复用**：布局组件只写一次，所有页面共用
- **权限集中管理**：路由守卫统一校验，避免重复代码
- **懒加载**：按需加载组件，提升首屏性能
