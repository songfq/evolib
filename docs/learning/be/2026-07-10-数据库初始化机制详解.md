# 2026-07-10 — 数据库初始化机制详解

---

## 一、问题背景

后端启动后，数据库中自动创建了表并插入了种子数据。需要理解：
1. 表是如何自动创建的？
2. 字段变更是否会自动同步？
3. DataInitializer 是什么，为什么会自动执行？

---

## 二、自动创建表的原理

### 2.1 配置触发

在 [application.yml](file:///e:/ai_fq/evolib/code/evolib-backend/src/main/resources/application.yml#L15-L18) 中配置了：

```yaml
spring:
  sql:
    init:
      mode: always                    # 始终执行初始化
      schema-locations: classpath:schema.sql  # 指定SQL文件路径
```

### 2.2 执行流程

```
Spring Boot 启动
    │
    ├── 加载 application.yml
    │       │
    │       └── 读取 spring.sql.init.mode=always
    │
    ├── 创建 DataSource（建立数据库连接）
    │
    ├── 查找 schema-locations 指定的文件
    │       │
    │       └── 找到 classpath:schema.sql
    │
    ├── 执行 schema.sql 中的 SQL 语句
    │       │
    │       ├── DROP TABLE IF EXISTS（删除旧表）
    │       ├── CREATE TABLE（创建新表）
    │       └── CREATE INDEX（创建索引）
    │
    └── 继续启动其他组件（Controller、Service 等）
```

### 2.3 mode 参数的可选值

| 值 | 含义 |
| --- | --- |
| `always` | 每次启动都执行 SQL 文件（开发环境） |
| `never` | 从不执行（生产环境） |
| `embedded` | 只对内存数据库生效 |

## 三、字段变更是否自动同步？

### 3.1 结论：不会

`schema.sql` 是**静态 SQL 文件**，只有修改这个文件并重启，表结构才会更新。

### 3.2 不同 ORM 框架的对比

| 框架/方式 | 字段变更是否自动同步 | 原理 |
| --- | --- | --- |
| **schema.sql（当前方式）** | ❌ 不会 | SQL 是静态的，需要手动修改 |
| **JPA / Hibernate** | ✅ 会 | 根据实体类自动生成 DDL |
| **Flyway / Liquibase** | ✅ 会 | 通过迁移脚本管理版本 |

### 3.3 如果需要添加字段

**当前方式**：
1. 修改 `schema.sql`，添加新字段
2. 重启后端

**示例**：给 books 表添加 `publisher` 字段

```sql
-- 修改前
CREATE TABLE IF NOT EXISTS books (
    isbn             VARCHAR(20)   PRIMARY KEY,
    title            VARCHAR(200)  NOT NULL,
    author           VARCHAR(100)  NOT NULL,
    ...
);

-- 修改后
CREATE TABLE IF NOT EXISTS books (
    isbn             VARCHAR(20)   PRIMARY KEY,
    title            VARCHAR(200)  NOT NULL,
    author           VARCHAR(100)  NOT NULL,
    publisher        VARCHAR(100),  -- 新增字段
    ...
);
```

## 四、以后执行会重新创建表吗？

### 4.1 结论：会

因为当前配置：
1. `mode: always` — 每次启动都执行 schema.sql
2. schema.sql 开头有 `DROP TABLE IF EXISTS` — 每次都删除旧表

### 4.2 后果

每次重启都会**清空所有数据**！包括：
- 之前注册的读者
- 已经借阅的记录
- 所有审计日志

### 4.3 解决方案

**开发环境**（允许重建）：
```yaml
spring:
  sql:
    init:
      mode: always
```

**生产环境**（禁止重建）：
```yaml
spring:
  sql:
    init:
      mode: never  # 或者不配置，默认就是 never
```

**改进方案**（保留数据）：
```sql
-- 不使用 DROP TABLE，只创建不存在的表
CREATE TABLE IF NOT EXISTS books (...);
CREATE TABLE IF NOT EXISTS readers (...);
-- 这样不会删除已有数据，但也不会修改表结构
```

## 五、DataInitializer 详解

### 5.1 什么是 DataInitializer

它是一个实现了 `CommandLineRunner` 接口的 Spring 组件：

```java
@Component
public class DataInitializer implements CommandLineRunner {
    
    @Override
    public void run(String... args) throws Exception {
        // 启动时执行的初始化逻辑
    }
}
```

### 5.2 为什么需要这个类

**核心原因**：密码需要 BCrypt 加密，不能直接写在 SQL 文件中。

```java
// DataInitializer 中（正确做法）
reader.setPasswordHash(passwordEncoder.encode("138001"));

// schema.sql 中（错误做法，明文密码不安全）
INSERT INTO readers (password_hash) VALUES ('138001');
```

**其他原因**：
- 需要动态生成数据（如当前时间、随机数）
- 需要调用 Spring 的其他 Bean（如 PasswordEncoder）
- 需要条件插入（只在表为空时插入）

### 5.3 执行逻辑

```java
@Override
public void run(String... args) throws Exception {
    if (bookMapper.selectCount(null) == 0) {
        initBooks();      // 只有 books 表为空时才插入
    }
    if (readerMapper.selectCount(null) == 0) {
        initReaders();    // 只有 readers 表为空时才插入
    }
    if (auditLogMapper.selectCount(null) == 0) {
        initAuditLogs();  // 只有 audit_logs 表为空时才插入
    }
}
```

**关键**：使用 `selectCount(null) == 0` 判断，避免重复插入数据。

### 5.4 其他替代方式

| 方式 | 优点 | 缺点 |
| --- | --- | --- |
| **DataInitializer**（当前） | 支持 BCrypt 加密，灵活控制 | 需要写代码 |
| **data.sql** | 简单，纯 SQL | 无法加密密码 |
| **Flyway/Liquibase** | 专业，支持版本管理 | 学习成本高 |
| **手动插入** | 最灵活 | 启动后需要手动操作 |

#### 方式二：data.sql

Spring Boot 也支持 `data.sql` 文件，会在 `schema.sql` 之后执行：

```yaml
spring:
  sql:
    init:
      schema-locations: classpath:schema.sql
      data-locations: classpath:data.sql
```

但无法加密密码，不推荐。

#### 方式三：Flyway（专业方案）

```java
// Flyway 会自动按版本号顺序执行迁移脚本
// db/migration/V1__create_tables.sql
// db/migration/V2__insert_data.sql
// db/migration/V3__add_publisher_column.sql
```

适合生产环境的数据库版本管理。

## 六、为什么会自动执行 DataInitializer

### 6.1 Spring Boot 生命周期机制

```
Spring 容器初始化
    │
    ├── 扫描所有 @Component 注解的类
    │       │
    │       └── 发现 DataInitializer
    │               │
    │               └── 发现实现了 CommandLineRunner 接口
    │
    ├── 实例化 DataInitializer
    │       │
    │       └── 注入依赖（BookMapper、ReaderMapper、PasswordEncoder 等）
    │
    ├── 容器启动完成（所有 Bean 都已创建）
    │       │
    │       └── 自动调用所有 CommandLineRunner.run()
    │               │
    │               └── 执行 seed data 插入
    │
    └── 应用启动完成
```

### 6.2 关键注解

| 注解/接口 | 作用 |
| --- | --- |
| `@Component` | 告诉 Spring 这是一个组件，需要被管理 |
| `CommandLineRunner` | 告诉 Spring 容器启动后执行 `run()` 方法 |

### 6.3 类似接口

| 接口 | 执行时机 |
| --- | --- |
| `CommandLineRunner` | 容器启动后，接收命令行参数 |
| `ApplicationRunner` | 容器启动后，接收 ApplicationArguments |
| `@PostConstruct` | Bean 初始化完成后（早于 CommandLineRunner） |

---

## 七、总结

| 组件 | 作用 | 执行时机 |
| --- | --- | --- |
| `schema.sql` | 创建表结构（DDL） | 数据源初始化后，容器启动前 |
| `DataInitializer` | 插入种子数据（DML） | 容器启动完成后 |
| `spring.sql.init.mode` | 控制是否执行 schema.sql | 启动时读取 |

### 当前配置的优缺点

✅ **优点**：
- 开发环境一键启动
- 数据自动初始化
- 无需手动操作

❌ **缺点**：
- 每次重启清空数据
- 不适合生产环境
- 表结构变更需要手动修改 SQL 文件

### 生产环境建议

1. 将 `spring.sql.init.mode` 改为 `never`
2. 使用 Flyway 或 Liquibase 管理数据库迁移
3. 种子数据手动插入或通过 SQL 脚本执行
